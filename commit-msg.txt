Fix router: add basename "/gag2-wiki" so home route matches

Bug: navigating to https://haritsnaufalich.github.io/gag2-wiki/
rendered the NotFoundPage in the <Outlet />, with the navbar
and footer rendering correctly above and below. The layout
was fine; the routed content was wrong.

Root cause: createBrowserRouter had no `basename` option.
React Router reads window.location.pathname, which is
"/gag2-wiki/", and tries to match it against the route
config. The parent route is `path: "/"` (no basename), so
the router treats "/gag2-wiki/" as a CHILD path. The home
route is `index: true` (matches only when URL is exactly
"/"), and the child paths are "crops", "pets", etc. None
match "/gag2-wiki/", so it falls through to `path: "*"`
and renders NotFoundPage.

Fix: pass `{ basename: "/gag2-wiki" }` as the second arg
to createBrowserRouter. The router strips the basename
from the URL before matching. So:
  URL "/gag2-wiki/"     -> matched as "/"     -> index: true (HomePage)
  URL "/gag2-wiki/crops" -> matched as "/crops" -> path: "crops" (CropsPage)
  URL "/gag2-wiki/crops/carrot" -> matched as "/crops/carrot" -> path: "crops/:slug"
  URL "/gag2-wiki/broken" -> matched as "/broken" -> path: "*" (NotFoundPage)

Link components automatically prefix the basename, so
<Link to="/crops"> renders to "/gag2-wiki/crops" — no
other code changes needed.

Bundle: 295.11 -> 295.29 kB (+0.18 kB, the basename
string). Build clean, 0 TS errors.

Why this wasn't caught earlier: my smoke tests checked
the HTML shell (has #root, has main JS) but never the
client-side route. The bug was always there since the
hash-router -> browser-router migration in commit 68ede62;
D14 in MEMORY.md documents the subpath as locked, but
the basename was never wired up.

Also documents this in the code as an inline comment so
the next maintainer doesn't strip it.
